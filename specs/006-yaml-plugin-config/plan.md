# Implementation Plan: YAML-Driven Agent Plugin Configuration

**Branch**: `006-yaml-plugin-config` | **Date**: 2026-02-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-yaml-plugin-config/spec.md`

## Summary

Overhaul the agent plugin system to support YAML configuration files that define agent commands, parameters, and metadata declaratively. The system auto-generates tool handlers from config, eliminating boilerplate code. Apollo is migrated as the reference implementation. Code-based plugins (Arachne) continue to work alongside config-based ones.

## Technical Context

**Language/Version**: Python 3.10+
**Primary Dependencies**: mcp>=1.26.0, mythic>=0.2.10, pydantic>=2.0.0, pyyaml>=6.0.0
**Storage**: N/A (stateless — config files read at startup)
**Testing**: pytest (unit tests, integration tests)
**Target Platform**: Linux server (MCP server process)
**Project Type**: Single project
**Performance Goals**: Plugin loading completes in <1s for 10 agents with 20 commands each
**Constraints**: Must maintain backward compatibility with existing MCP tool schemas
**Scale/Scope**: ~20 agent commands per config file, ~5 agents maximum

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle                      | Status | Notes                                                                                     |
| ------------------------------ | ------ | ----------------------------------------------------------------------------------------- |
| I. MCP Protocol First          | PASS   | Tools registered via same MCP interface; config-driven tools are indistinguishable         |
| II. Async-Native Architecture  | PASS   | Generated handlers call async execute_with_validation; no blocking I/O                    |
| III. Plugin Isolation           | PASS   | Config-based plugins load independently; failures skip the plugin, don't crash the server  |
| IV. Explicit Authorization     | PASS   | Tool descriptions from config; callback_id required; agent type validated at execution     |
| V. Fail-Safe Defaults          | PASS   | Invalid configs rejected at startup with specific errors; unknown types rejected            |

**Post-Phase 1 Re-check**: All principles still satisfied. YAML config loading uses safe_load (no code execution). Dynamic Pydantic model creation is deterministic from validated config data.
